# TripZone 백엔드 AI 작업 브리프

이 문서는 백엔드 팀원이 ChatGPT, Claude, Gemini 같은 AI에 그대로 넣어서 현재 프로젝트 맥락을 빠르게 먹일 수 있게 만든 브리프다.

## 1. 이 프로젝트가 무엇인지

- 프로젝트명: `TripZone`
- 성격: 국내 숙소 예약 서비스 팀 프로젝트
- 현재 기준: `프론트 우선`
- 백엔드 목표: 설계도 기준으로 Oracle DB, API, 인증, 상태 흐름을 맞춰 프론트와 연결 가능하게 구현
- 루트 패키지 기준: `com.kh.trip`

핵심 도메인:

- 회원/권한
- 판매자 승인
- 숙소/객실
- 예약
- 결제
- 리뷰
- 이벤트/쿠폰
- 마일리지
- 문의
- 관리자 운영

## 2. 가장 중요한 규칙

이 프로젝트의 source of truth는 현재 돌아가는 백엔드 코드가 아니라 `docs/` 설계 문서다.

우선순위는 아래 순서로 본다.

1. `docs/tripzone-company-spec.md`
2. `docs/tripzone-structure-spec-v2.md`
3. `docs/tripzone-ddl-v2.sql`
4. `docs/02_기능명세서.md`
5. `docs/06_프론트연동가이드.md`

즉, 현재 백엔드 구현이 설계도와 다르면 기본적으로 `설계도 쪽을 맞는 기준`으로 본다.

## 3. 프론트 현재 상태

프론트는 화면 mock 단계에서 끝난 상태가 아니라, 백엔드가 바로 이어붙이기 쉽게 한 번 정리된 상태다.

현재 프론트 구조:

- `frontend/src/lib/appClient.js`
  - 공통 데이터 소스 진입점
- `frontend/src/lib/mockDb.js`
  - localStorage 기반 mock DB
- `frontend/src/services/`
  - `lodgingService.js`
  - `bookingService.js`
  - `mypageService.js`
  - `dashboardService.js`
- `frontend/src/pages/`
  - 라우트 진입 화면
- `frontend/src/features/`
  - 뷰모델, UI 블록, 유틸

중요:

- 지금 백엔드 연결은 `pages`를 직접 뜯는 방식이 아니라 `services`를 API 호출로 교체하는 방식으로 진행해야 한다.
- 프론트는 이미 `appClient -> services -> mockDb/data` 경계로 정리돼 있다.

## 4. 백엔드가 AI에게 받아야 하는 작업 방향

AI에게 백엔드 작업을 시킬 때 목표는 아래다.

- 설계도 기준으로 도메인/상태값/DTO를 정리
- Oracle DDL 기준으로 엔티티와 시퀀스를 맞춤
- 프론트가 기대하는 응답 shape를 서비스/DTO에서 맞춤
- 인증, 예약, 결제, 문의, 관리자 API를 연결 가능한 상태로 만듦

하면 안 되는 것:

- 설계도에 없는 새 기능 추가
- 프론트 화면 흐름을 임의로 바꾸라고 제안
- 문의 도메인을 단순화해서 설계도보다 축소
- `com.kh.trip` 외 패키지 기준으로 새 구조를 벌림

## 5. 설계도 기준으로 반드시 맞춰야 하는 항목

### 5.1 문의

문의는 설계도 기준으로 아래 구조가 맞다.

- `InquiryRoom`
- `InquiryMessage`

문의 핵심 흐름:

- 문의방 생성
- 관련 숙소/예약 연결
- 회원/판매자/관리자 메시지 송수신
- 읽음 여부
- 상태 관리

문의 상태값 기준:

- `OPEN`
- `ANSWERED`
- `CLOSED`
- `BLOCKED`

현재 백엔드가 단일 `Inquiry` 엔티티나 `PENDING`, `COMPLETED`, `DELETE` 같은 상태를 쓰고 있다면 설계도 미반영 상태로 본다.

### 5.2 관리자 운영

설계도 기준으로 관리자 영역은 단순 조회가 아니라 운영 API가 있어야 한다.

필요 영역:

- 판매자 승인 관리
- 회원 상태 관리
- 이벤트/쿠폰 운영
- 문의 모니터링
- 감사 로그 조회

`AdminController`가 비어 있거나 운영 API가 없다면 미완성 상태다.

### 5.3 Oracle 기준

DB는 Oracle 기준으로 맞춘다.

- PK: `NUMBER(10)` + 시퀀스
- 문자열: `VARCHAR2`
- 날짜 기본값: `SYSDATE`
- 등록/수정일: `REG_DATE`, `UPD_DATE`
- soft delete 필요 시: `DELETED_AT`

실제 기준 원문은 `docs/tripzone-ddl-v2.sql`이다.

## 6. 현재 백엔드와 프론트가 충돌하기 쉬운 지점

### 6.1 인증

프론트 mock 기준:

- `email`
- `password`

일부 현재 백엔드 구현 기준:

- `loginId`
- `password`

판정:

- 인증은 설계 문서가 세부 필드명까지 완전히 고정한 건 아니라 `합의 필요 항목`이다.
- 한 번 합의하면 프론트 service와 백엔드 DTO를 그 기준으로 고정하면 된다.

### 6.2 숙소 DTO

프론트가 화면에서 쓰는 대표 키:

- `lodgingId`
- `lodgingName`
- `lodgingType`

현재 백엔드는 `lodgingNo` 같은 명명일 수 있다.

이 경우:

- 백엔드 DTO를 바꾸거나
- 프론트 `services/lodgingService.js`에서 어댑터로 매핑

둘 중 하나로 처리한다.

### 6.3 예약 DTO

프론트가 기대하는 핵심 값:

- `bookingId`
- `bookingStatus`
- `bookingStatusLabel`
- `lodgingId`
- `roomId`
- `checkInDate`
- `checkOutDate`
- `guestCount`
- `totalPrice`

현재 백엔드가 `bookingNo`, `roomNo`를 써도 된다.
다만 프론트 연결 시 `services/bookingService.js`에서 최종 화면 shape를 맞춰줘야 한다.

### 6.4 결제 DTO

프론트가 기대하는 핵심 값:

- `paymentId`
- `bookingId`
- `paymentStatus`
- `paymentStatusLabel`
- `paymentAmount`
- `payMethod`
- `approvedAt`

결제는 백엔드 DTO가 프론트 mock보다 더 실전형이어도 괜찮다.
핵심은 프론트 서비스에서 최종 키와 상태 라벨을 맞추는 것이다.

## 7. 백엔드 작업 우선순위

### 1순위

- 인증 DTO/응답 구조 확정
- 숙소 목록/상세 API
- 예약 생성/조회 API
- 결제 생성/조회 API

### 2순위

- 리뷰 API
- 마이페이지 요약/목록 API
- 판매자 숙소/예약 운영 API

### 3순위

- 문의 `InquiryRoom/InquiryMessage` 구조 반영
- 관리자 운영 API
- 감사 로그

## 8. 프론트와 연결할 때 건드릴 파일

프론트 팀과 협업할 때 우선 보면 되는 파일:

- `frontend/src/lib/appClient.js`
- `frontend/src/lib/mockDb.js`
- `frontend/src/services/lodgingService.js`
- `frontend/src/services/bookingService.js`
- `frontend/src/services/mypageService.js`
- `frontend/src/services/dashboardService.js`
- `docs/06_프론트연동가이드.md`

중요:

- 프론트 `pages/`와 `features/`는 이미 많이 정리돼 있어서, API 연동은 우선 `services` 교체 방식으로 가는 게 맞다.

## 9. AI에게 바로 넣을 문장

아래 문장을 그대로 복붙해서 AI에 넣으면 된다.

```md
너는 TripZone 백엔드 구현을 돕는 시니어 스프링 백엔드 엔지니어다.

프로젝트는 국내 숙소 예약 서비스이고, source of truth는 현재 백엔드 코드가 아니라 설계 문서다.
반드시 아래 우선순위를 따른다.

1. docs/tripzone-company-spec.md
2. docs/tripzone-structure-spec-v2.md
3. docs/tripzone-ddl-v2.sql
4. docs/02_기능명세서.md
5. docs/06_프론트연동가이드.md

기술 기준:
- 패키지 루트는 com.kh.trip
- DB는 Oracle
- PK는 NUMBER(10)+시퀀스
- 상태값은 설계 문서 문자열 기준

프론트 연결 기준:
- 프론트는 appClient -> services -> mockDb/data 구조다.
- 프론트 pages를 먼저 뜯지 말고 services 기준으로 API 연결 가능하게 맞춘다.
- 프론트는 lodgingService, bookingService, mypageService, dashboardService를 기준으로 연결한다.

중요 도메인 규칙:
- 문의는 InquiryRoom + InquiryMessage 구조가 맞다.
- 문의 상태는 OPEN, ANSWERED, CLOSED, BLOCKED가 맞다.
- 관리자 운영 API는 설계도에 있는 범위까지 구현해야 한다.
- 인증은 email/loginId 중 하나로 먼저 합의하고 DTO를 고정해야 한다.

작업 원칙:
- 설계도에 없는 기능은 추가하지 마라.
- 현재 코드가 설계도와 다르면 설계도 기준으로 수정 방향을 제시하라.
- 프론트가 기대하는 응답 shape를 DTO와 서비스 계층에서 맞춰라.
- 가능하면 엔티티를 그대로 노출하지 말고 화면/도메인 기준 DTO를 제안하라.

먼저 해야 할 일:
1. 현재 백엔드 구현과 설계 문서의 충돌 항목을 정리
2. 우선순위 API 목록 제시
3. 필요한 DTO/엔티티/컨트롤러/서비스 수정안 제시
4. Oracle 테이블/시퀀스 기준 점검
5. 프론트 services와 연결될 응답 예시 JSON 제시

답변은 추상적으로 하지 말고, 실제 클래스명/DTO명/상태값/API 경로 수준으로 적어라.
```

## 10. 백엔드 팀원이 AI에게 같이 주면 좋은 파일

최소 이 파일들과 함께 넣는 게 좋다.

- `docs/tripzone-company-spec.md`
- `docs/tripzone-structure-spec-v2.md`
- `docs/tripzone-ddl-v2.sql`
- `docs/06_프론트연동가이드.md`

이미 구현이 있다면 추가로:

- 현재 `controller/`
- 현재 `dto/`
- 현재 `domain/`
- 현재 `service/`

## 11. 최종 판단 기준

AI가 좋은 답을 하고 있는지 판단하는 기준은 아래다.

- 설계도 기준을 먼저 잡는가
- Oracle/시퀀스/상태값 기준을 놓치지 않는가
- 문의를 `InquiryRoom/InquiryMessage`로 이해하는가
- 프론트 연결을 `services` 기준으로 보게 만드는가
- `숙소/예약/결제/리뷰`와 `문의/관리자`를 같은 방식으로 섞어 처리하지 않는가
- DTO, API 경로, 상태값을 실제 구현 가능한 수준으로 말하는가

이 기준을 벗어나면 맥락을 잘못 먹은 답변으로 본다.
